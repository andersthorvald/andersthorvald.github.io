4 个概念,一句话讲透

1 / 4

SUID

"临时拿主人身份证办事" — 执行时进程变成属主

chmod u+s file # 字母法 chmod 4755 file # 数字法
  • 前提: 执行者命中 owner / group / others 哪一列,那一列必须有 x
  • 大小写提示: rws 小写 = 有 x + SUID 生效;rwS 大写 = SUID 摆设
  • 风险: 越少越安全,审计命令: find / -perm -4000
2 / 4

SGID

"子子孙孙继承组" — 团队目录必备,新文件自动归组

chmod g+s /project chmod 2775 /project
  • 作用对象: 目录(让新文件继承组)
  • 隐藏特性: 子目录自动继承 SGID 位本身,多层目录组继承不断层
  • 配合权限: 属组需要 w 才能创建文件,如 drwxrwsr-x
3 / 4

Sticky Bit

"公共厕所规则" — 大家都能用,只能拆自己建的

chmod +t /tmp chmod 1777 /tmp
  • 作用范围: 只保护目录项操作(删/重命名),不管文件内容读写
  • 经典: /tmp 的 drwxrwxrwt,最后那个 t 就是它
  • root 例外: root 依然能删任何人的文件(运维清场需要)
4 / 4

chattr +i

"连 root 也得先解锁" — 终极文件锁

chattr +i /etc/passwd # 上锁 chattr -i /etc/passwd # 解锁 lsattr /etc/passwd # 查属性
  • 关键: 不是特殊权限,是文件属性。需要 CAP_LINUX_IMMUTABLE capability
  • 容器注意: Docker / K8s 默认 drop 掉这个 capability,连 root 也解不开
  • 配合 auditd: 监控 chattr 调用,谁解锁谁留痕

6 道面试题 · 关键洞察

Q1

SUID -rwsr-xr-x,alice 执行时进程是什么身份?命中哪一列?

以 root 身份运行;alice 命中 others 列(她不是 root,不在 root 组),others 是 r-x,有 x 所以能执行,SUID 生效。

SUID 生效的硬性前提:执行者命中的那列必须有 x。

Q2

SGID chmod g+s /project 后,A 在里面建 a.txt 和子目录 subdir/,subdir/ 有 SGID 位吗?

有。子目录自动继承 SGID 位本身,这是 SGID 隐藏特性。否则多层目录继承就断了,只有第一层有效。

SGID 继承是位本身也传下去,不只是组属性。

Q3

Sticky A 在 /tmp/secret.sh 设 777,B 能 cat / echo >> / rm 它吗?

cat ✅ 能(others 有 r);echo >> ✅ 能(others 有 w);rm ❌ 不能。Sticky Bit 只保护目录项操作(删/重命名),不管文件内容读写。

/tmp 里大家互相看得到内容,但删不掉别人的文件 — 这是 Sticky 而非"加密"。

Q4

实战 www-data 被入侵想提权,第一条 SUID 审计命令?

find / -perm -4000 -type f 2>/dev/null。重点看:异常位置(/tmp、/var/tmp 下) + 近期修改 + 属主是 root 的可疑程序。

进阶还要查 SGID(-2000)和 capability(getcap -r /)。

Q5

深度 为什么 SUID 脚本比 SUID 二进制危险得多?(2 个原因)

① 脚本是文本,有写权限就能篡改,SUID 位还在下次执行就是攻击者代码;② 依赖环境变量(PATH、IFS、LD_PRELOAD、BASH_ENV),可被劫持执行任意代码。

再加 2 个:解释器历史漏洞多 + execve 到解释器打开脚本之间存在 race condition。

Q6

综合 保护 /etc/passwd 不被篡改,至少做哪几层防护?

① chmod 644 DAC 最小权限;② chattr +i 上锁;③ auditd 监控 chattr / passwd 命令;④ AIDE / Tripwire 文件完整性;⑤ 定期备份;⑥ 集中认证(LDAP/AD);⑦ 容器化 readOnlyRootFilesystem。

"防御深度"是核心思路 — 任何单层都能被绕过,多层叠加才能扛住。

5 个常见误区

  • ❌ "SGID 是设给文件的,Sticky 是设给目录的" SGID 也可设文件(强制 lock 顺序等场景);Sticky 历史上对文件还有"保留内存"含义,现代基本只用目录。日常理解可以这样记,但别当绝对。
  • ❌ "chmod 4755 后,任何用户都自动变 root 跑" 前提是执行者命中的那列必须有 x。否则 SUID 完全不生效,直接 Permission denied。
  • ❌ "Sticky Bit 让 /tmp 里的文件互相不可见" Sticky Bit 只管删/重命名,不管读/写。默认 644 时,大家能读;777 时大家能改内容。只是不能"销毁"。
  • ❌ "chattr +i 设上后 root 也改不了" root 能改,只是必须先 chattr -i 留痕。配合 auditd 才能抓住。容器里 root 没 CAP_LINUX_IMMUTABLE 时才是真改不了。
  • ❌ "SUID/SGID/Sticky 可以同时设" 可以,chmod 7755 file = SUID + SGID + 755。但通常没必要,组合场景少见,理解单个含义更重要。
